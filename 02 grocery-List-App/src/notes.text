This line:

const updatedGroceryList = [...groceryItems];

is making a shallow copy of the groceryItems array and storing it in a new variable called updatedGroceryList.

Let's break it down:

Step 1: groceryItems

Suppose you have:

const groceryItems = [
  { name: "Milk" },
  { name: "Eggs" },
];

This array lives somewhere in memory.

Step 2: ...groceryItems

The spread operator (...) takes every element out of the array and "spreads" them out.

...groceryItems

becomes

{ name: "Milk" },
{ name: "Eggs" }

Think of it like saying:

"Take every element from this array."

Step 3: [ ...groceryItems ]

The square brackets create a new array containing those elements.

So

[...groceryItems]

is equivalent to writing

[
  groceryItems[0],
  groceryItems[1]
]

The result is:

const updatedGroceryList = [
  { name: "Milk" },
  { name: "Eggs" }
];
What does this accomplish?

Now you have two different arrays.

groceryItems !== updatedGroceryList

This is true because they're different arrays.

However...

groceryItems[0] === updatedGroceryList[0]

This is also true because they both point to the same object.

This is why it's called a shallow copy.

Visual representation
groceryItems
      │
      ▼
+-----------------------+
|  ○  |  ○             |
+-----------------------+
   │      │
   ▼      ▼
 {Milk}  {Eggs}

updatedGroceryList
        │
        ▼
+-----------------------+
|  ○  |  ○             |
+-----------------------+
   │      │
   └──────┘
 (same objects)

Notice:

The arrays are different.
The objects inside are the same.
Why is this commonly used in React?

React expects you not to mutate state directly.

Instead of doing:

groceryItems.push(newItem); // ❌ modifies the original array

you do:

const updatedGroceryList = [...groceryItems];
updatedGroceryList.push(newItem);

Now you've created a new array, which React can detect as a state change.

Excellent question. This gets to the heart of how React detects state changes.

You're correct that updatedGroceryList is only a shallow copy. So why not just do this?

groceryItems.push({
  name: inputValue,
  quantity: 1,
  completedStatus: false,
});

SetGroceryItems(groceryItems);

The answer is because React compares the array reference, not its contents.

Let's look at what happens in memory.

Initially:

groceryItems
     │
     ▼
Array A

Now if you write:

groceryItems.push(newItem);
SetGroceryItems(groceryItems);

The array is still Array A.

groceryItems
     │
     ▼
Array A  (same array, just one more element)

So React sees:

"The old state was Array A."

"The new state is... Array A."

Nothing changed by reference, so React may skip re-rendering because it thinks the state is the same.

Now look at your code.

const updatedGroceryList = [...groceryItems];

This creates

groceryItems ----------► Array A

updatedGroceryList ----► Array B

Array B initially contains the same objects, but it is a different array.

When you call

SetGroceryItems(updatedGroceryList);

React sees

Old state: Array A
New state: Array B

Those references are different.

React knows the state changed and re-renders.

But wait...

You noticed this line:

updatedGroceryList[indexItem].quantity++;

This does modify the object from the old state because the objects are shared between the two arrays.

That's why this is only a shallow copy.

In small demo apps you'll often see this, but in larger React apps it's considered mutating state, which React recommends avoiding.

A more immutable approach would be to create a new object for the changed item as well.

So the purpose of updatedGroceryList is not to protect the objects.

Its purpose is to create a new array reference.

Think of it like this:

[...] → gives React a new array ✔️
It does not give you new objects inside that array ❌
